Graphs

Definitions

  • A graph or network is defined by a set of vertices and a set of edges consisting of ordered or unordered pairs of vertices from . Many problems have already been solved for graphs. The skill to develop is to cast data into a graph representation and pick the right already known algorithm for graphs. Examples are city maps, electronic circuits, social media and so forth...
  • A graph is undirected if edge implies that is also in . Road networks between cities are typically undirected. Street networks within cities are almost always directed because of one-way streets. Another examples are “callgraphs” where each node is a routine and edges represent a routing that calls another routine. This can help to find “dead code”: unconnected piece of code.
  • In weighted graphs, each edge (or vertex) of is assigned a numerical value, or weight. The edges of a road network graph might be weighed with their length, drive-time or speed limit. In unweighted graphs, there is no cost distinction between various edges and vertices.
  • Certain types of edges complicate the task of working with graphs. A self loop and a multi edge (edge that occurs more than once between two edged) make the graph non simple.

  • A graph is sparse when only a small fraction of the possible number of vertex pairs actually have edges defined between them.
  • A graph is dense if each node can have a connection to any other node (considering that direction also matters). Then the number of edges is with the number of nodes. The efficiency of an algorithm on graphs will be evaluated on the number of nodes and edges, and often case algorithms perform best on sparse graphs. E.g road graphs are generally sparse.
  • An acyclic graph does not contain any cycles. Trees are connected acyclic undirected graphs.
  • DAGs are Directed Acyclic Graphs.

  • In labeled graphs, each vertex is assigned a unique name or identifier to distinguish if from all the other vertices. An important graph problem is isomorphism testing, determining whether the topological structure of two graphs are in fact identical if we ignore any labels.

Representing a Graph

Recall that a graph contains vertices and edges. A graph can be represented with different data structures, namely

Adjacency Matrix

We can represent using an matrix , where elements is 1 if is an edge of and 0 if it isn't. It may use excessive space for graphs with many vertices and relatively few edges, however. Can we save space if the graph is (1) undirected? (2) sparse ? Yes, in this case we only need to store the upper triangular part of the adjacency matrix, as the connection to each node’s predecessors are already there.

Adjacency List

An adjacency list consists of a array of pointers, where the ith element points to a linked list of the edges incident on vertex i.

To test if edge is in the graph, we search the ith list for which takes where is the degree of the ith vertex. Note that can be much less than n when the graph is sparse.

  • Test if exists is just as easy as looking to an entry in the adjacency matrix.
  • Find vertex out-degree (connection) is equivalent to sum up the row in the matrix and getting the ith list size .
  • When graphs are sparse, lists are obvious winners
  • When graphs are dense, lists come with all the overhead of pointers storage and matrices are minor winner.

Both representations are very useful and have different properties, although adjacency lists are probably better for most problems. In any case we want to define a structure containing node information and the graph.

We will talk about many algorithms that can be applied on graph to solve a set of different problems. In particular we will see the following ones:

  • Auxiliary data structures

Before going deeper in the single algorithms implementation we will make a small detour on two topics which will be used in the rest of the section. Many algorithms on graphs traverse it and along the way they build a tree (they try not to connect nodes that have already been visited, as this would be a waste of processing time and lead to unless loop or recursion). Therefore we need a way to store this tree along the way. In other different problems, we might want to add nodes to different subgraphs and from time to time efficiently find if the node already belongs to a subgraph and merge the two subgraph (set of nodes) together. This is generally done using a clever data structure called union-find (as it supports the union and find methods).

Auxiliary Data Structures

Representing Trees: the Parent array

A very simple way to represent a tree is to store an array with as many entries as the nodes. Entry i stores the index of its parent. To traverse the tree we then see we loop for each entry and traverse the child to parent relationship until we hit and entry with nullptr (or -1 if referring to indices) as parent. In the meanwhile we need to keep a visited array to make sure we don't reprocess (e.g print) parents already visited as predecessors of other nodes.

Topological sorting

Topological sorting is what allows us to provide an ordering of a direct acyclic graph (DAG). A topological sort of a graph is an ordering on the vertices so that all edges go from left to right. Consider for example the dressing schedule (how we get dressed in the morning). We can model the precedence constraints on the different clothes items as a DAG.

It is not possible to obtain a topological ordering in a undirected graph, since there is no notion of precedence in such a graph. Similarly, if there is a loop, we cannot find a topological sort of the graph since there is a cyclic dependency. Note that candidate starting nodes for a topological sort are nodes that have in degree (no parents). Therefore we keep a indegree order and in a consecutive manner pop nodes which have degree 0. As this are popped, new nodes will have indegree zero and will be pushed to the queue. We continue until the queue is empty.

Minimum Spanning Tree (MST)

A tree is a connected graph with no cycles. A spanning tree is a subgraph of which has the same set of vertices of and is a tree. A minimum spanning tree of a weighted graph is the spanning tree of whose edges sum to minimum weight. There can be more than one minimum spanning tree in a graph → consider a graph with identical weight edges for example.

Minimum spanning tree is always taught in algorithm courses since:

Example application 1: Minimum spanning trees are useful in constructing networks, by describing the way to connect a set of sites using the smallest total amount of wire. Example application 2: Minimum spanning trees provide a reasonable way for clustering points in space into natural groups.

Prims Algorithm

Prims algorithm starts from one vertex and grows the rest of the tree an edge at a time. As a greedy algorithm it picks the cheapest edge with which can grow the tree by one vertex without creating a cycle. 

prims_minimum_spanning_tree(graph G)
  - Select an arbitrary vertex (s) to start the tree from
  - while (there are still non-tree vertices)
      - select the edge of minimum weight between the tree and non selected vertices    // greedy choice
      - add the selected edge and vertex to the tree T_prim

We start from A and we add the edge of weight 1. After we choose the minimum edge available to the grown tree, which is the edge 1. We repeat and we choose now 3 ((there are many so we break tie somehow, e.g. based on node labels) which is the minimum attached to the already grown tree non creating cycles and so forth ... The question is, how do we know that this procedure produces the minimum spanning tree?

Why is Prim correct?

We use a proof by contradiction: given a starting minimum spanning tree, the algorithm cannot extend to a node x making the tree non MST. If this would be the case there could be another path reaching out from the current tree to the node x with smaller weight, but then the path leading the node x and passing so to speak through node v has a edge to v which is less then the direct edge to x. But Prims would choose this edge instead, therefore we reach a contradiction. More detailed proof here: https://home.uncg.edu/cmp/faculty/srtate/330.f16/primsproof.pdf

Implementation

Implementation of this algorithm, requires to know which is the newest minimum available edge on the fringe (unattached border to the tree being grown so far). We can implement prims algorithm in 2 ways:

  • storing a vector of distances that we update as we add new border nodes, and all the time pick a node in this list not yet in the tree and with minimum distance
  • storing distances of the grown tree to every new vertex in a heap. That means that for every edge that changes, we need to update the corresponding heap.

Implementation with distances vector 

int mst_prim_dist(graph *g, int start) {
  int i;                               /* counter */
  edge_node *p;                        /* temporary pointer */
  bool intree[MAX_VERTICES];           /* is the vertex in the tree yet? */
  int distance[MAX_VERTICES];          /* cost of adding to tree */
  int v;                               /* current vertex to process */
  int w;                               /* candidate next vertex */
  int dist;                            /* cheapest cost to enlarge tree */
  int weight = 0;                      /* tree weight */
  
  // initialization of flags
  for (i = 1; i <= g->nvertices; i++) {
    intree[i] = false;
    distance[i] = MAXINT;
    parent[i] = -1;
  }
  
  distance[start] = 0;
  v = start;
  while (!intree[v]) { // n
    intree[v] = true;
    if (v != start) {
      weight = weight + dist;
    }
    
    // wwe are adding a new node -> update nodes distances if this is smaller then the one found so far
    // we will go over each child of the node which is O(n)
    p = g->edges[v];
    while (p != nullptr) {
      w = p->y;
      if ((distance[w] > p->weight) && (!intree[w])) {
        distance[w] = p->weight;
        parent[w] = v;
      }
      p = p->next;
    }

    // in the previous loop we have updated distances of a newly added node, pick the one which is not yet in the tree and is the minimum
    dist = MAXINT;
    for (i = 1; i <= g->nvertices; i++) {   // O(n)
      if ((!intree[i]) && (dist > distance[i])) {
        dist = distance[i];
        v = i;
      }
    }
   // total of one while loop O(n)+O(n)=O(n). This will be executed n times. 
  }
  return(weight);
}

Kruskall’s Algorithm

The tree here does not need to grow from the same root, but multiple trees can grow and they connect as they go. At every iteration we try to add the smallest edge to one of the growing trees. If the edge to be added (the minimum) would create a cycle, we discard it.

  • We have to go through all edges from smallest to largest. Thus, we need to sort them. This will take (as for example using the heap sort algorithm). Since is at most it follows that and therefore this will take .
  • Once we choose an edge, we need to ask the question whether the new edge to add to the tree creates a cycle. We can store the currently found tree into separate sets. We can store nodes in a data structure, where each node alone is a trivial 1-element tree. As we add an edge between two nodes belonging to two different sets, we have to find the nodes (find to which set they belong) and then the two sets become a new set given by the sum of the two (union). The data structure storing this sets and supporting the union and find operations is called Union-Find. If x and y are in the same set, then they create a cycle, otherwise they do not.
